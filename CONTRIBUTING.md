# Contributing to CoStruct

Thanks for contributing to CoStruct.

This repository is currently documentation-first.
CoStruct does not aim to become a software codebase.
Contributions should strengthen methodology and documentation, not turn the repository into an implementation project.

Most contributions at this stage will improve:

- methodology clarity
- conceptual consistency
- examples and case structure
- documentation quality
- reusable collaboration patterns

## Before You Contribute

Please align with the current project direction:

- CoStruct is not a prompt dump
- CoStruct is not a general AI news repository
- CoStruct is not a random tooling list
- CoStruct is building a reusable methodology for human-AI collaboration

If a change adds content but weakens structure, it is probably the wrong change.

## What Kind of Contributions Are Useful

Useful contributions usually fall into one of these categories:

- clarify an existing concept
- identify a conceptual conflict or ambiguity
- improve wording without flattening the core judgments
- add a better example or case structure
- propose a reusable collaboration pattern
- improve validation or accumulation mechanisms
- fix mistakes, typos, broken links, or unclear references

## Preferred Contribution Style

Prefer small, focused changes.

Good:

- one conceptual correction
- one terminology cleanup
- one document restructuring pass
- one new example with clear purpose

Avoid bundling unrelated changes into one pull request.

## Ground Rules

When contributing, try to preserve these principles:

- state is prior to task
- deeper AI intervention requires stronger validation
- maturity is measured by accumulation, not only by one-shot output
- humans keep direction, judgment, tradeoffs, and responsibility
- AI can help explore, clarify, generate, analyze, review, and accelerate

## Writing Guidelines

When editing documents:

- keep the writing direct
- prefer strong structure over vague inspiration
- avoid inflated terminology unless it adds real precision
- do not replace clear judgments with soft generic prose
- do not turn methodology documents into prompt collections

If you introduce a new term, define it clearly and keep it distinct from nearby concepts.

Examples:

- `state` is not the same as `role`
- `validation` is not the same as `accumulation`
- `intervention depth` is not the same as `task complexity`

## Issues

Use issues for:

- bug reports in repository content
- conceptual inconsistencies
- methodology proposals
- unclear definitions
- structural problems in documentation

When opening an issue, be concrete.
Point to the exact sentence, section, or document whenever possible.

## Pull Requests

When opening a pull request:

- explain what changed
- explain why the change is needed
- explain whether it changes meaning, wording, or structure
- keep scope tight

If your change affects methodology terms or structural judgments, call that out explicitly.

## Commit Messages

Preferred examples:

- `docs: refine state recognition section`
- `docs: clarify intervention depth definition`
- `docs: add methodology proposal template`
- `chore: polish contribution workflow`

## Discussion Standard

CoStruct prefers rigorous disagreement over vague agreement.

If you think a concept is wrong, say:

- what is wrong
- where it breaks
- what distinction is missing
- what alternative structure is better

Critique is useful when it improves clarity, boundaries, and repeatability.
